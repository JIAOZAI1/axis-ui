# 企业级前端设计规范方案

> 版本 v1.0 · 适用范围:企业内部所有 Web 中后台、门户及移动 H5 项目

---

## 一、方案总览

本方案的目标是建立一套**可扩展、可维护、多端一致**的设计语言体系,解决企业多产品线视觉割裂、开发与设计沟通成本高、主题定制困难三个核心问题。

整体架构分为四层,自下而上依次为:

```
┌─────────────────────────────────────────────┐
│  L4 业务模板层   页面模板 / 业务区块(Pro Blocks)  │
├─────────────────────────────────────────────┤
│  L3 组件库层     Button / Table / Form 等 UI 组件 │
├─────────────────────────────────────────────┤
│  L2 设计规范层   色彩 / 字体 / 间距 / 动效等视觉规则 │
├─────────────────────────────────────────────┤
│  L1 设计 Token 层  原子变量,设计与代码的唯一事实源  │
└─────────────────────────────────────────────┘
```

设计 Token 是整个体系的地基:设计师在 Figma 中维护 Token,通过工具链自动同步到代码仓库,前端不再硬编码任何视觉值。

## 二、设计 Token 体系

### 2.1 三层 Token 架构

Token 采用业界主流的三层结构(参考 Salesforce Lightning、Ant Design、Adobe Spectrum 的实践):

| 层级 | 名称 | 说明 | 示例 |
|------|------|------|------|
| 第一层 | 全局 Token(Global / Primitive) | 原始值池,不带语义,仅描述"有什么" | `blue-500: #1677FF`、`space-4: 16px` |
| 第二层 | 语义 Token(Semantic / Alias) | 描述"用在什么场景",引用全局 Token | `color-primary: {blue-500}`、`color-text-secondary: {gray-600}` |
| 第三层 | 组件 Token(Component) | 组件级微调,引用语义 Token | `button-primary-bg: {color-primary}`、`table-header-bg: {color-fill-quaternary}` |

**关键约束:**

1. 业务代码和组件样式**只允许消费语义 Token 和组件 Token**,禁止直接引用全局 Token 或硬编码色值;
2. 换肤、暗色模式只需替换语义 Token 与全局 Token 的映射关系,组件层零改动;
3. 每个 Token 必须有 description 字段,说明使用场景。

### 2.2 Token 命名规范

命名采用 `[类别]-[属性]-[场景]-[状态/层级]` 的结构,全小写、连字符分隔:

```
color-text-primary          主要文字色
color-text-secondary        次要文字色
color-bg-container          容器背景色
color-border-default        默认描边色
color-primary-hover         品牌色悬浮态
space-component-md          组件内中等间距
radius-card                 卡片圆角
shadow-overlay              浮层阴影
motion-duration-fast        快速动效时长
```

在不同技术栈中的表现形式保持一一对应:

| 载体 | 形式 | 示例 |
|------|------|------|
| Figma Variables | 斜杠分组 | `color/text/primary` |
| CSS Custom Properties | 变量,带团队前缀 | `--acme-color-text-primary` |
| JS/TS Token 包 | 驼峰常量 | `colorTextPrimary` |
| 小程序/原生端 | 由 Style Dictionary 编译生成 | 各平台格式 |

### 2.3 Token 源文件格式

Token 源文件采用 W3C Design Tokens 社区规范(DTCG)的 JSON 格式,存放于独立仓库 `@acme/design-tokens`:

```json
{
  "color": {
    "blue": {
      "500": { "$value": "#1677FF", "$type": "color" }
    },
    "text": {
      "primary": {
        "$value": "{color.gray.900}",
        "$type": "color",
        "$description": "一级文字,用于标题与正文主体"
      }
    }
  }
}
```

通过 **Style Dictionary**(或 Tokens Studio + CI)编译输出多端产物:CSS 变量、Less/Sass 变量、TS 常量、iOS/Android 资源文件。

## 三、统一视觉方案

### 3.1 色彩系统

**品牌色板生成规则**:以品牌主色为基准,通过 HSV 梯度算法(可直接采用 Ant Design 色板算法或 Radix Colors 的 12 级体系)生成 10 级色阶,保证同一色相下明度、饱和度过渡自然。

```
主色示例(以科技蓝为例,可替换为企业 VI 主色):
blue-1  #E6F4FF   浅色背景
blue-2  #BAE0FF   浅色描边
blue-3  #91CAFF   
blue-4  #69B1FF   
blue-5  #4096FF   悬浮态
blue-6  #1677FF   ★ 主色(默认态)
blue-7  #0958D9   激活态
blue-8  #003EB3   
blue-9  #002C8C   
blue-10 #001D66   
```

**功能色**固定四类,含义在全公司范围内不可挪用:

| 语义 | 色相基准 | 典型用途 |
|------|---------|---------|
| Success 成功 | 绿 `#52C41A` | 成功提示、通过状态 |
| Warning 警告 | 橙 `#FAAD14` | 警告提示、待处理 |
| Error 错误 | 红 `#FF4D4F` | 错误提示、危险操作、必填标记 |
| Info 信息 | 蓝(同主色或独立蓝) | 中性通知 |

**中性色**采用带透明度的方案(如 `rgba(0,0,0,0.88)`),而非实色灰阶,好处是在任意底色上叠加都能保持层次正确,且天然适配暗色模式:

```
color-text-primary      rgba(0,0,0,0.88)   一级文字
color-text-secondary    rgba(0,0,0,0.65)   二级文字
color-text-tertiary     rgba(0,0,0,0.45)   三级文字/占位符
color-text-disabled     rgba(0,0,0,0.25)   禁用文字
color-border-default    rgba(5,5,5,0.15)   默认描边
color-border-split      rgba(5,5,5,0.06)   分割线
color-fill-hover        rgba(0,0,0,0.04)   悬浮填充
color-bg-layout         #F5F5F5            页面底色
color-bg-container      #FFFFFF            容器底色
color-bg-elevated       #FFFFFF            浮层底色
```

**无障碍要求(硬性)**:正文文字与背景对比度 ≥ 4.5:1(WCAG AA),大号文字与图标 ≥ 3:1。Token 发布流水线中加入自动对比度校验,不达标即阻断发布。

### 3.2 字体系统

**字体族**:

```css
--font-family-base: -apple-system, BlinkMacSystemFont, "Segoe UI",
  Roboto, "Helvetica Neue", Arial, "PingFang SC",
  "Hiragino Sans GB", "Microsoft YaHei", sans-serif;
--font-family-code: "SF Mono", Menlo, Consolas, "Courier New", monospace;
--font-family-number: /* 数据大屏可选用等宽数字字体,如 tabular-nums 特性 */;
```

**字号阶梯**:以 14px 为基准(中后台阅读密度的最优解),按接近 1.2 的比率生成阶梯,行高对应取整为偶数:

| Token | 字号 | 行高 | 用途 |
|-------|------|------|------|
| font-size-xs | 12px | 20px | 辅助说明、标签 |
| font-size-sm | 13px | 22px | 表格密集数据(可选) |
| font-size-base | 14px | 22px | 正文默认 |
| font-size-lg | 16px | 24px | 强调正文、小标题 |
| font-size-h4 | 20px | 28px | 卡片标题 |
| font-size-h3 | 24px | 32px | 区块标题 |
| font-size-h2 | 30px | 38px | 页面标题 |
| font-size-h1 | 38px | 46px | 门户/营销页大标题 |

**字重**只保留三档:Regular 400(正文)、Medium 500(强调、标题)、Semibold 600(数据数字、大标题),避免多字重导致的渲染不一致。

### 3.3 间距与布局

**间距基于 8px 网格,辅以 4px 半步长**:

```
space-1: 4px    space-2: 8px    space-3: 12px   space-4: 16px
space-5: 20px   space-6: 24px   space-8: 32px   space-10: 40px
space-12: 48px  space-16: 64px
```

使用约定:组件内部间距用 4–16px,组件之间用 16–24px,区块之间用 24–48px。禁止出现网格之外的魔法数字(如 13px、17px)。

**栅格与断点**:24 列栅格,断点与容器规则如下:

| 断点 Token | 视口宽度 | 典型设备 |
|-----------|---------|---------|
| screen-xs | < 576px | 手机 |
| screen-sm | ≥ 576px | 大屏手机 |
| screen-md | ≥ 768px | 平板 |
| screen-lg | ≥ 992px | 小尺寸笔记本 |
| screen-xl | ≥ 1200px | 桌面(中后台设计基准 1440px) |
| screen-xxl | ≥ 1600px | 大屏显示器 |

中后台页面以 1440×900 为设计基准稿,内容区最小适配到 1200px。

### 3.4 圆角、阴影与层级

**圆角**按组件体量分档,全公司统一:

```
radius-sm: 4px    小组件(Tag、Checkbox)
radius-md: 6px    常规控件(Button、Input、Select)
radius-lg: 8px    容器(Card、Modal、Popover)
radius-full: 9999px  胶囊 / 头像
```

**阴影**与 z-index 层级绑定,阴影只用于表达"漂浮",不用于装饰:

| Token | 值(简化示意) | z-index | 用途 |
|-------|------|---------|------|
| shadow-sm | 0 1px 2px rgba(0,0,0,.06) | — | 卡片轻微抬升 |
| shadow-md | 0 4px 12px rgba(0,0,0,.08) | 1000 | Dropdown / Popover |
| shadow-lg | 0 8px 24px rgba(0,0,0,.12) | 1010 | Modal / Drawer |
| shadow-overlay | 0 12px 32px rgba(0,0,0,.16) | 1030 | 全局通知、Tour |

z-index 全部收敛为 Token(`z-dropdown: 1000`、`z-modal: 1010`、`z-message: 1020`、`z-tooltip: 1030`),业务代码禁止手写 z-index。

### 3.5 动效规范

动效遵循「快进快出、有始有终」原则,只在状态变化、层级出入场、反馈确认三类场景使用:

```
motion-duration-fast: 0.1s     悬浮、按压等微反馈
motion-duration-mid:  0.2s     开关、Tab 切换
motion-duration-slow: 0.3s     Modal、Drawer 出入场

motion-ease-out:     cubic-bezier(0.215, 0.61, 0.355, 1)   入场
motion-ease-in:      cubic-bezier(0.55, 0.055, 0.675, 0.19) 出场
motion-ease-in-out:  cubic-bezier(0.645, 0.045, 0.355, 1)   位移/形变
```

必须响应系统的 `prefers-reduced-motion`,该设置开启时所有装饰性动效降级为直接切换。

### 3.6 图标与插画

图标统一使用单一图标库(自建 SVG 图标库或基于 Iconfont/Lucide 定制),规格约束:24×24 网格、2px 描边、线性风格为主,面性风格仅用于选中态。图标以 SVG 组件形式通过 npm 包分发,颜色继承 `currentColor` 以便随文字色自动换肤。空状态、引导页插画建立统一插画库,色板取自设计 Token,禁止各团队自行外采风格不一的素材。

## 四、暗色模式与多主题

### 4.1 实现机制

主题切换完全通过替换 CSS 变量作用域实现,组件与业务代码无感知:

```css
:root { /* 亮色映射 */
  --acme-color-bg-container: #ffffff;
  --acme-color-text-primary: rgba(0, 0, 0, 0.88);
}
[data-theme="dark"] { /* 暗色映射 */
  --acme-color-bg-container: #1f1f1f;
  --acme-color-text-primary: rgba(255, 255, 255, 0.85);
}
```

暗色模式的要点:不是简单反色,而是重建一套映射——背景从 `#141414` 到 `#2a2a2a` 分 4 级表达海拔层次(海拔越高越亮),品牌色降低饱和度约一档(如 blue-6 → blue-5 的明度)避免刺眼,阴影在暗色下弱化、改用描边和背景亮度差表达层级。

### 4.2 品牌主题定制

子品牌 / 私有化客户定制时,只需提供一份「主题包」(覆盖全局 Token 中的品牌色 + 少量语义 Token),经色板算法自动派生完整色阶,其余 Token 继承默认值。主题包同样走 Token 仓库的发布流程,保证可追溯。

## 五、工程落地方案

### 5.1 技术选型建议

| 环节 | 方案 |
|------|------|
| Token 源管理 | Figma Variables + Tokens Studio 插件,JSON 存 Git |
| Token 编译 | Style Dictionary,输出 CSS var / Less / TS / 移动端产物 |
| 组件库 | 基于 Ant Design 5(其本身就是 Token 化架构)做企业主题封装,或自研(React + CSS Variables) |
| 样式方案 | CSS Variables 为核心;业务侧可搭配 CSS Modules 或 Tailwind(将 Tailwind 配置指向 Token) |
| 文档站 | Storybook / dumi,含组件 Demo、Token 速查表、设计指引 |
| 质量卡口 | Stylelint 自定义规则禁止硬编码色值/间距;CI 中跑视觉回归(Chromatic / Playwright 截图对比) |

### 5.2 Token 同步流水线

```
Figma Variables(设计师维护)
      │  Tokens Studio 导出 / API 拉取
      ▼
tokens 仓库(JSON, PR 评审 + 对比度自动校验)
      │  CI: Style Dictionary 编译
      ▼
@acme/design-tokens npm 包(语义化版本)
      │
      ├──▶ @acme/ui 组件库(消费语义/组件 Token)
      ├──▶ 各业务前端项目(CSS 变量注入)
      └──▶ 移动端资源文件
```

版本策略:新增 Token 为 minor,修改 Token 值为 patch(视觉微调)或 minor(明显变化),删除/重命名 Token 为 major 且必须提供 codemod 迁移脚本与至少一个版本的废弃期(deprecated 警告)。

### 5.3 业务侧使用约定

```tsx
// ✅ 正确:消费语义 Token
const Card = styled.div`
  background: var(--acme-color-bg-container);
  border-radius: var(--acme-radius-lg);
  padding: var(--acme-space-6);
  box-shadow: var(--acme-shadow-sm);
`;

// ❌ 禁止:硬编码 & 引用全局 Token
const Bad = styled.div`
  background: #fff;
  border-radius: 8px;
  color: var(--acme-blue-500); /* 全局 Token 不允许直接消费 */
`;
```

## 六、治理与推广

规范能否落地,治理机制比规范本身更重要。建议建立以下机制:

**组织**:成立虚拟的设计系统小组(2–3 名设计师 + 2–3 名前端),负责 Token 与组件库的评审、发布和答疑;各业务线设一名「大使」负责规范落地反馈。

**流程**:任何新增 Token 或组件需求走统一的 RFC 流程——提案(说明场景与复用度)→ 双周评审会 → 试验性发布(alpha)→ 转正。业务线不满足于现有组件时,先在业务仓库孵化,复用度达到 2 个以上项目后再"晋升"进入公共组件库。

**度量**:以三个指标衡量健康度——Token 覆盖率(业务代码中硬编码视觉值的占比,目标 < 5%)、组件复用率、设计走查缺陷密度。通过 Stylelint 报告与代码扫描定期出数。

**文档与培训**:文档站是唯一权威来源,包含每个 Token 的用途说明与正误示例;每季度对新人做一次设计系统培训;重大版本升级配套迁移指南与 codemod。
